# 29 雲端 DB 瘦身計畫(WP-M3R,2026-07-14 規劃)

> **狀態:Planner 提案,尚未執行**。本檔記錄 2026-07-13夜間~2026-07-14 一次即時事故排查(法人/分點資料斷層 → 查出 `radar.db` 已達 7.35GB,GitHub Release 備份逼近 2GB 硬上限)延伸出的完整規劃。取代 `docs/26` WP-M3 原本「只拆分點」的假設——實測發現分點資料不是主要肥胖來源。
> 本檔完成前,`docs/26` WP-M3/WP-M4 的容量估算(`<1.5GB`、`+3-6GB`)已知過時,以本檔數字為準。
> 依 `docs/17` 高風險流程,本檔是 Planner 提案(第1步),尚未經 Reviewer 審查與使用者正式核准動工,不得依此直接改 schema/搬移資料。

## 0. 事故起因(背景,非本檔重點)

2026-07-13 使用者發現網站資料斷層(大盤07-13、法人卡07-09、分點卡07-08)。排查後發現：VPS 已完成 top500×490天分點歷史深度回補並上傳,導致雲端 `radar.db` 從安全的 ~1GB 暴增到 **7.35GB**,GitHub Release 單檔 2GB 上限、Actions cache 10GB 上限雙雙逼近,曾在更早之前造成過 `EOFError`/`database disk image is malformed` 的真實事故(已知教訓,見 `AGENTS.md` 危險清單)。搶修過程中另外燒光了當月 GitHub Actions 2,000 分鐘免費額度,已由使用者將 repo 改為 **public**(解除限制,public repo 無 Actions 分鐘數上限)。

## 1. 實測:7.35GB 真正的組成(`dbstat` 直接查證,非估算)

| 表(含索引) | 大小 | 列數 | 佔比 |
|---|---|---|---|
| **`indicators_daily`**(技術指標) | **~3.96GB** | 9,927,956 | **~52%** |
| `daily_prices`(股價) | ~1.19GB | 10,135,778 | ~16% |
| `branch_trades`(分點) | ~1.53GB(索引占1.03GB,幾乎是資料本身2倍) | 7,190,518 | ~21% |
| `warrant_daily`(權證) | ~0.90GB | 10,205,842 | ~12% |
| 其餘小表 | <0.1GB | — | <1% |

合計 ≈7.46GB,與實測 7.35GB 吻合(誤差在合理範圍內,確認沒有其他隱藏肥胖來源)。

**關鍵修正**:原本 `docs/26` WP-M3 假設分點資料是主要問題,實測分點只佔21%,**技術指標表 `indicators_daily` 才是最大宗、超過分點2.5倍**。只拆分點無法真正解決 2GB 上限風險。

## 2. 逐表判決

### 2.1 `indicators_daily`(技術指標)—— ✅ 可大幅瘦身,使用者已確認(2026-07-14)

**查證結論**:資料庫裡的歷史指標列**沒有任何功能會讀取**——
- 前端 K 線圖的均線/BOLL/MACD/RSI/KD 疊圖是 `web/components/KChart.tsx` **用股價即時在瀏覽器端算出來的**,不是讀資料庫的歷史指標值。
- `pipeline/radar/export/json_export.py` 只在匯出「今天」這一天的技術面板(分數、觸發理由)時讀這張表(`ti.date = :d`),從不讀歷史列。
- 原始設計文件 `docs/05_database_schema.md` 本來就寫「可重算,只留2年」,只是從未真的實作。

**使用者關切澄清**:瘦身**不影響**「股票自上市以來的完整走勢圖(K線本體)」——那張圖來自 `daily_prices`(股價原始表),此次規劃**明確排除、永遠保留全歷史**,兩張表是分開的。使用者已確認理解並同意可刪減 `indicators_daily` 歷史。

**空間分解**:每列約372 bytes,其中約200 bytes(~53%)是每天的觸發理由/風險 JSON 文字(`reasons`/`risks`),歷史的理由文字沒有任何功能顯示,只有「今天」的會顯示在個股頁。

**建議策略**:只保留最近 **400 個交易日**(約1.6年,兼顧 Phase 3 策略績效回測需要的 S1-S10 全市場標記歷史),其餘刪除。

**預估效果**:3.96GB → **~0.4GB**。

### 2.2 `daily_prices`(股價)—— ❌ 不動,維持全歷史

**查證結論**:個股頁 K 線圖「全部」時間選項,雲端**每天(不是每週)匯出 JSON 時都會讀該股票上市以來的完整股價history**(`json_export.py` 榜單聯集股票無 LIMIT 讀全歷史)。這跟分點資料只有 VPS 週更需要全歷史的情況完全不同——股價全歷史是雲端每日必需品。

**其他理由**:只佔DB 16%,且股價資料無法免費重抓(FinMind 額度限制),風險/效益比不划算。

**維持**:~1.19GB,不變。

### 2.3 `branch_trades`(分點)—— 延續 `docs/26` WP-M3,但修正設計

**新發現**:索引比資料本身還肥(索引1.03GB vs 資料535.5MB)——因為 `branch_name`(分點名稱文字)直接進索引鍵,831個不重複分點名稱卻在719萬列裡重複存文字。

**建議策略**(兩件事一起做):
1. **正規化**:把 `branch_name`/`broker_id` 抽成小的對照表(`branch_dim`,僅831×64種組合),主表改存數字 `branch_id`,既有讀取邏輯用 VIEW 相容包裝,不必逐一改13處讀取程式碼。單此項預估省 41%(1.53GB→~0.90GB)。
2. **保留窗口**:雲端只留近 **130個交易日**(比原案120天多10天緩衝,對應個股頁「籌碼日報」120日功能),更舊資料搬移至 VPS 常駐 `branch_hist.db`,週快照備份至 Cloudflare R2(沿用 `docs/21` 現成R2設計)。

**預估效果**:1.53GB → **~0.22GB**(全市場擴容後估~1.1GB)。

### 2.4 `warrant_daily`(權證)—— 補實作原本規劃過但沒做的保留政策

**查證結論**:`docs/05` 原本就寫要保留2年,但清理排程從未真的實作(`docs/STATUS.md` 已知債務有記錄)。目前每年增速約1,000萬列。

**發現一個需要先修的 bug**:`aggregate_warrants()` 沒指定日期時會**整表刪光重建** `warrant_stock_daily`,若貿然砍 `warrant_daily` 舊資料,重建時會連帶把已算好的彙總歷史也弄丟——這個 bug 必須在瘦身前修好。

**建議策略**:只保留近 **150個交易日**(比 docs/05 原本的2年短,但足夠涵蓋現有所有讀取功能,含120日權證大戶追蹤窗)。

**預估效果**:0.90GB → **~0.55GB**。

## 3. 瘦身後總量預估

**現在(全市場擴容前)**:1.19(股價)+ 0.4(指標)+ 0.22(分點)+ 0.55(權證)+ ~0.1(其他)≈ **2.45GB**,壓縮後估 **~0.9GB**——離2GB上限有兩倍以上緩衝。

**全市場擴容後(WP-M2+WP-M4上線)**:約 **3.35GB**,壓縮後估 **~1.1GB**,仍安全。此數字取代 `docs/26` WP-M3 原本「<1.5GB」的過時目標。

## 4. 兩個必須先修的既有程式碼問題(瘦身前置,否則瘦身效果會被打回原形)

1. **`compute_indicators()` 全歷史重算模式會把所有歷史指標列重新寫回資料庫**——VPS 手動重算鏈(`vps_backfill_plan.md` Step 4c/4d)與雲端 `task=adjust`/`indicators-only` 都會觸發。瘦身前必須先把這個全歷史模式改成「只寫保留窗口內的資料」,否則下次手動重算又把 `indicators_daily` 撐回3.96GB。
2. **`aggregate_warrants()` 沒帶日期參數時會整表清空重建**——瘦身 `warrant_daily` 前必須先修這個,不然重建過程會連帶洗掉已經算好的彙總資料。

## 5. 建議分階段執行順序

```
Phase 0(程式碼修正,低風險,可先做):
  a. compute_indicators 全歷史模式改為只寫保留窗口(§4-1)
  b. 修正 aggregate_warrants 無日期時的整表重建邏輯(§4-2)
  c. 新增 `radar prune` CLI 指令(指標400日/權證150日/import_logs 180日;分點窗口留給 Phase 2)

Phase 1(指標+權證瘦身,風險低,效益最大):
  VPS:git pull → radar prune → VACUUM → integrity_check → 上傳 release + 清 Actions cache(沿用 vps_backfill_plan.md 4e 既有流程)
  雲端:daily-branches.yml 加一步週期性 prune(不動 WAL/cache/release 續存鏈本身)
  效果:7.35GB → ~3.3GB,壓縮~1.0GB——立即解除2GB上限的急迫風險

Phase 2(= docs/26 WP-M3 修正版,高風險,需完整走 docs/17 核准流程):
  分點正規化(branch_dim)+ 130日窗口 + branch_hist.db 常駐VPS + R2週快照 + 還原演練

Phase 3(= docs/26 WP-M2 + WP-M4,依賴 Phase 2):
  WP-M2:daily-branches.yml 改全市場單輪抓(17:40輪全市場,21:00輪拿掉分點爬蟲只留融資券+算分數,避免 Actions 分鐘數雙倍)
  WP-M4:VPS 全市場×2年分點歷史回補(排除ETF,估~8.6天;範圍1,991檔股票,現有1,267檔已有部分資料、1,129檔完全沒有歷史需從頭抓)
```

Phase 0+1 刻意排在 Phase 2 之前:先解除2GB上限的急迫壓力,讓後續分點高風險搬遷不用趕時間做。Phase 1 與 WP-M4 互不阻塞。

## 6. 風險清單

| # | 風險 | 對策 |
|---|---|---|
| R1 | 舊版程式碼重算會撐回肥胖(見§4) | Phase 0 必須先於 Phase 1 完成並確認 VPS/雲端都已 `git pull` |
| R2 | `aggregate_warrants` 整表重建洗掉彙總歷史 | Phase 0b 修正為前提 |
| R3 | 個股頁「籌碼日報」240日功能,分點窗口只留130天會截斷 | 使用者需決定:接受截斷+UI誠實標註,或窗口拉到240天(多花~0.9GB) |
| R4 | 雲端每日 `compute-branch-stats`(daily-branches.yml)在分點窗口縮小後,排行榜統計會只反映短窗口,偏離全歷史統計 | 需在 Phase 2 前決定:改成只在VPS週更(ATTACH全歷史),雲端不再每日重算 |
| R5 | daily_prices 保持不動,是本計畫刻意決定,任何「順手也砍股價表」的建議應被拒絕 | 明確排除,不在後續 Reviewer/Executor 範圍內 |
| R6 | 分點正規化(branch_dim)寫入邏輯若有 bug,可能造成同一分點產生重複ID | 需要 seed test 驗證新舊查詢結果一致(parity test) |
