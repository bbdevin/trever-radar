> 注意:各家額度與方案會變動,實作前以官網當下條款為準;本文標註為 2026 年中認知。

## 2. 各來源深評

### TWSE / TPEx 官方

**適合**:一切盤後日級資料——日K(`MI_INDEX` 全市場單一請求)、法人(`T86`)、融資券、注意/處置、權證主檔與每日成交。穩定、權威、免費、無認證。
**不適合/坑**:
1. 分點日報(`bsr.twse.com.tw`)**每檔一請求 + CAPTCHA**,全市場 ~2000 檔爬不動也不該爬;只能當自選股備援。TPEx 分點頁較鬆但同樣逐檔。
2. 無盤中逐筆(僅 5 秒級揭示,且不建議高頻打)。
3. 上市(TWSE)與上櫃(TPEx)API 格式完全不同,欄位、日期格式(民國年)都要各寫 parser——Provider 層要吸收這件事。
4. 偶爾改版無預警 → 每日健檢必要。
5. 授權:公開資料個人研究使用低風險;**對外散布或商用需簽資訊使用契約**。≤10 人私用可接受,但不要公開網址。

### FinMind

**適合**:歷史回補(日K、還原價、法人、融資券可一次拉多年)、**分點資料的最務實來源**(全市場、結構化、免爬蟲)、V3 回測資料、新聞。
**不適合**:盤中訊號。其即時資料為快照級(約 10 秒),做不了逐筆大單與主動買賣判定;免費層有請求數限制(註冊約 600 次/小時級),回補歷史要排隊慢慢拉。
**判定**:你的理解正確。定位 = 盤後主力來源之一 + 歷史回補 + 分點資料付費來源。贊助方案每月數百元,換來全案最關鍵的資料,**這筆錢建議直接花**,比自建爬蟲便宜且穩。
**風險**:單一民間服務,若停服則分點資料斷炊 → 備援:bsr 逐檔抓自選股 + 監控池(≤50 檔可行),以及本地永久落地所有已抓資料。

### Fugle MarketData

**適合**:V2 盤中。WebSocket 提供逐筆成交(含內外盤方向可推斷主動買賣)、1 分 K、五檔;REST 有快照可低頻輪詢大範圍。文件品質好、有官方 Node/Python SDK。
**免費版**:連線數與同時訂閱檔數很有限(數檔級)、REST 每分鐘請求數有限 → 適合開發期與「只盯 top 5~10 檔 + 其餘用快照輪詢」的混合策略。
**付費開發者方案**(每月數百元級):訂閱檔數擴到足以覆蓋 50~200 檔監控池。
**判定**:你的理解正確。V2 起步 = 免費版驗證訊號邏輯 → 好用再升級。設計上把「逐筆訂閱名單」限制成監控池前段(依盤後分數排序取前 N),其餘池內股票用 1 分鐘 REST snapshot 輪詢,兩軌並行,額度永遠可控。
**不適合**:盤後歷史(有但非強項)、分點(無)。

### 元大 SPARK API

**適合**:V3、你個人:查庫存、帳務、損益;手動確認後的下單頁(人按按鈕才送單)。
**不適合/風險**:
1. 憑證與帳號綁定個人,**絕不可**讓其他 9 位使用者透過你的憑證看到或操作任何帳務 → 必須獨立服務、獨立開關、僅 admin 可見。
2. 自動下單風險(程式 bug = 真金白銀)→ 本專案永久不做自動下單;下單功能即使做,也要二次確認 + 每日額度上限。
3. 券商 API 條款通常禁止再散布行情 → 不要拿它當行情源餵給全系統。
4. 綁死風險:全系統不得 import 元大 SDK,只有獨立的 `PersonalBroker` 模組可以。
**判定**:列 V3、個人專用、查詢優先,正確。

### 其他考慮過的來源

- **神秘金字塔、籌碼K線等第三方網站爬蟲**:資料整理得好,但違反其使用條款、隨時被封,不採用。
- **Shioaji(永豐)**:API 成熟、有逐筆,但你非永豐戶;若未來 Fugle 不敷使用可評估開戶,列為潛在替代。
- **公開資訊觀測站(MOPS)**:公司基本資料、工廠/公司地址(做地緣對照用)、除權息;V1 用到地址與除權息。

## 3. 各版本資料源配置

- **V1**:TWSE + TPEx(日級全套)+ FinMind(還原價、歷史回補、分點)。零盤中。
- **V2**:+ Fugle(免費→視需要付費)。
- **V3**:+ FinMind 深歷史與新聞、+ 元大 SPARK(個人)、(選)美股免費行情源(如 Stooq/yfinance 級)做題材連動。

## 4. 逐筆大單需要什麼資料(回答你的問題)

單筆大單 = 逐筆成交的 `價 × 量`。主動買賣 = 成交價貼近賣一(外盤)→ 主動買;貼近買一(內盤)→ 主動賣。因此需要**逐筆成交 + 當下五檔(或 tick 內含內外盤註記)**,Fugle WS 的 trades 頻道即含此資訊。FinMind 10 秒快照做不到單筆判定,只能做「10 秒內金額暴增」的粗訊號——這就是為什麼盤中主力源選 Fugle。

## 5. 分點與權證的盤中可行性(回答你的問題)

- 分點買賣超**沒有盤中公開資料**,一律 T+1 盤後。系統設計已據此:盤中只用「昨日分點分數」當底分,不假裝盤中知道分點動向。
- 權證盤中價量:Fugle 可訂權證代號,但監控池加權證會吃額度;V2 折衷 = 對 A 級訊號股票,臨時把其成交值前 2 大權證加入快照輪詢,確認「權證同步放大」後在理由中加註。V1 不做。

## 6. 可替換資料來源架構

### 介面(V1 就建,但薄)

```
DataProviders/
├─ Contracts/
│  ├─ DailyPriceProvider        # fetchDailyPrices(date): DailyPriceDTO[]
│  ├─ InstitutionalProvider     # 法人買賣超
│  ├─ MarginProvider            # 融資券
│  ├─ WarrantProvider           # 權證主檔+每日成交
│  ├─ BrokerTradeProvider       # 分點買賣超
│  ├─ AlertStockProvider        # 注意/處置
│  └─ IntradayProvider (V2)     # subscribeTrades/getSnapshot
├─ Twse/…  Tpex/…  FinMind/…  Fugle/… (V2)
└─ DTO/    # 統一欄位:date, stock_id, 數值欄…(內部格式,唯一真相)
```

### 規則

1. **DTO 是唯一內部格式**:民國年→西元、千股→張、格式差異全在各 Provider 內消化;資料庫只認 DTO。
2. **config 綁定**:`config/providers.php` 指定每種資料用哪個 Provider;換來源 = 改一行 config。
3. **來源記錄**:每張匯入表都有 `source` 欄 + `import_logs` 記錄(來源、日期、筆數、耗時、狀態)。
4. **品質檢查**(匯入後自動跑):筆數 vs 20 日均值 ±30%、關鍵股票(0050 成分)是否齊、數值合理性(價>0、量≥0)、日期正確。任一失敗 → 標記該日資料 `suspect` + 發告警,**評分排程照跑但清單標註「資料存疑」**。
5. **失效隔離**:單一 Provider 失敗自動 fallback 到備援 Provider(若有),否則跳過該資料並降級——缺分點就出「無分點版清單」並註明,系統不整體掛掉。
6. **V1 從簡**:每個 Contract 先只有一個實作 + FinMind 備援兩三個,不要為抽象而抽象;介面存在的目的是讓 V2/V3 換源不動下游。
